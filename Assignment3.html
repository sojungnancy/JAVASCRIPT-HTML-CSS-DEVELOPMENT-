Background

You are no doubt aware that Aotearoa New Zealand has three official languages: New Zealand Sign Language (NZSL, Te Reo Turi), Te Reo Māori, and English. We aim to design and develop a web application to learn NZSL, assisting beginners. The web app employs a service-oriented architecture, encompassing a UI front-end and a service back-end. The first two assignments asked you to contribute to constructing the service back-end, and this assignment continues your journey in helping to construct the front-end.

This assignment examines the separation of data from the UI, and how such data can be consumed and utilized in the end-user application. A subsequent assignment will delve into potential vulnerabilities that may surface when such an architecture is poorly implemented.

In the first two assignments, you implemented a handful of data sources that provide and update key information content. These data sources facilitate the separation of information content from its presentation.

Please read and understand the entire assignment before embarking on a solution.

NZSL Web Application

You are now ready to build the client application using the endpoints similar to those you developed in your first two assignments. The endpoints can be observed through the API documentation available at https://cws.auckland.ac.nz/nzsl/swagger/index.html which also allows you to experiment with the endpoints. Note that not all endpoints are used.

The application should display the logo supplied by one of the server endpoints at an appropriate place, and should consist of the following logical sections:

🏠 Home
This section is the landing place containing an introduction to the application. Display the version of the server at the bottom of the section.
📖 NZSL
This section contains selected NZSL signs to learn. The list could grow large, so you will need to have a search bar to dynamically shrink the list. There is an endpoint to support this.
📅 Events
This section shows to the user a list of events as supplied by the server. The user is able to add selected events to their calendar.
👪 User Registration
This section allows a user to register by choosing a username and password, and providing an address.
👩 User Login
This section allows a registered user to login. Access to this section is only granted if a user is not logged in.
✍ Guest Book
This section shows recent comments from the users. It also allows registered users to enter comments.

Add a favicon depicting the logo for the application.

Key Requirements

The application should use a combination of HTML, CSS, and JavaScript. Where the application needs icons, please consider using an appropriate Unicode character that represents the icon. For example, the icons used in this specification are all Unicode characters. We will look at another approach to constructing icons later on.

Please note the following requirements.

  1. You are not allowed to use any libraries or frameworks. (Why oh why?)
  2. The application should contain a single HTML page (named UPI.html where UPI is your personal login name, such as jbon007).
  3. The application should work on any modern browser (such as Chrome, Safari and Edge).
  4. The application should work on small screen devices. You can test this using the mobile view supported by the developer console of the browser. Note: you are not expected use CSS @media queries to style based on screen sizes. The mobile view requirement is to make sure you have viewed your application solution using the mobile view and considered your layout options based on the mobile view.
  5. There are submission requirements at the end of this document. Please read and understand the whole document before attempting a solution to the assignment.

This application should be of a professional quality to attract full marks. You should pay attention to details.

Test Plan

You need to think about how you would test and validate your application. To help you get started, here are a few points that might help you.

  1. Showing the correct logo
  2. Having the correct favicon for the application
  3. Having the logical sections (e.g., home and guestbook) and being able to see only one section at a time
  4. Being able to view the current set of comments
  5. Being able to submit comments
  6. Being able to see the list of signs
  7. Being able to search the list of signs
  8. Being able to shrink the list of signs dynamically
  9. Being able to see the images of signs
  10. Showing the current events in a readable format
  11. Being able to see the start/end dates in NZ time (not UTC time)
  12. Being able to download each calendar entry
  13. Being able to successfully register a user
  14. Showing the appropriate status message upon registration attempt
  15. Showing an appropriate error message if logging in is unsuccessful
  16. Login status (logged in or not) shown
  17. Logout option available and functional (when logged in)
  18. Post comment action re-directed to login if the user is not currently logged in

Discussing your test plan on the discussion forum with your peers will be of use, and is strongly encouraged.

Developing a test plan is an important aspect of software development. If you don't do a good one, you stand to produce low quality software which can put off your customers. Please imagine you are a tester, and pay attention to forming a good test plan. Not only you form a test plan, but also you execute it carefully.

Please test every little step you take before you go to the next step. In your browser, always keep the debugger panel open during your testing. Make sure that there are no bugs reported there. When there are issues, please read the error message carefully. Almost always these error messages tell you what the problem is. Please read and try to understand these messages. If there are server errors, chances are that you are sending in a malformed request. Again read the error message carefully, and this will shed some light into what you might be doing wrong.
